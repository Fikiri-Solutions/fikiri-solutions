# Postgres Migration Audit

> Goal: Make Fikiri's backend functionally and idiomatically Postgres-native across every feature, with no SQLite-isms left in runtime paths. No workarounds.

## Decisions (locked 2026-05-10)

| # | Decision | Detail |
|---|---|---|
| 1 | **Postgres-only runtime** | Drop SQLite code branches and SQLite-specific SQL from runtime modules. Tests and local dev move to a real Postgres (Docker / managed dev DB). |
| 2 | **Schema-as-is by default** | Do not change column types, constraints, defaults, or indexes without explicit approval. Behavioral fixes that don't touch schema can ship freely. Any column-shape change is paused for review. |
| 3 | **Branch + per-feature PRs** | Audit branch: `chore/pg-audit`. One PR per feature area (CRM, Email, Billing, Auth, Webhooks, Privacy, Automations, Analytics, Onboarding, Idempotency/RateLimit/Logging, Misc). |

## Scope

Runtime backend Python modules:

```
analytics/   core/   crm/   email_automation/   integrations/   routes/   services/
app.py       scripts/prestart.py
```

Out of scope (for now):

- Test files (`tests/`) — covered by Phase 3.
- One-off migration scripts (`scripts/migrations/`).
- Legacy stubs flagged `# deprecated` (frozen, no new imports).

## What this audit catches

Beyond the existing `scripts/check_postgres_sql_patterns.py` (which only flags 5 narrow patterns), this audit treats the following classes as bugs:

| Class | Pattern | Postgres-correct replacement |
|---|---|---|
| Upsert | `INSERT OR REPLACE`, `INSERT OR IGNORE` | `INSERT ... ON CONFLICT (...) DO UPDATE/NOTHING` |
| Auto-id | `INTEGER PRIMARY KEY AUTOINCREMENT` | `BIGSERIAL PRIMARY KEY` (or `GENERATED BY DEFAULT AS IDENTITY`) — **schema change, ask first** |
| Insert-id | `cursor.lastrowid`, `last_insert_rowid()` | `INSERT ... RETURNING id` |
| Booleans | `WHERE bool_col = 1/0` | `WHERE bool_col = TRUE/FALSE` (and stop storing booleans in `INTEGER` columns) |
| Datetimes from DB | `datetime.fromisoformat(row['col'])` | accept native `datetime` from psycopg2 |
| Datetime TEXT comparisons | `row['ts'] > timestamptz` | cast TEXT columns explicitly, or change schema to TIMESTAMP — **ask first** |
| SQLite date funcs | `datetime('now', ...)`, `julianday()`, `strftime()`, `date('now')` | `NOW()` / `CURRENT_TIMESTAMP` / `INTERVAL` arithmetic / `to_char()` / `EXTRACT()` |
| Null-handling | `IFNULL(a, b)` | `COALESCE(a, b)` |
| Concat | `'a' \|\| 'b'` | both work, but prefer `CONCAT(a, b)` for clarity with NULLs |
| Pattern match | `GLOB`, `REGEXP`, NOCASE `LIKE` | `LIKE`/`ILIKE`/`~`/`SIMILAR TO` |
| Catalog | `sqlite_master` | `information_schema.tables` / `pg_catalog.pg_tables` |
| Pragmas | `PRAGMA ...` | drop; equivalents are session settings or `SET` |
| Placeholders | `?` in raw `cursor.execute(...)` | go through `db_optimizer` (it adapts) — never raw `?` against psycopg2 |
| `DISTINCT` + `ORDER BY` | `SELECT DISTINCT ... ORDER BY col_not_in_select` | move to `GROUP BY` or include the ordering column in SELECT |
| `LIKE` placeholder escaping | literal `%` in SQL plus `?` placeholders | escape `%` → `%%` in the SQL before psycopg2 sees it (already handled in `core/postgres_compat.py`, but raw-SQL paths must use the adapter) |
| Stripe object leaks | returning `StripeObject`/`ListObject` to `jsonify` | flatten to plain `dict`/`list` (today's pattern) |
| Type strictness | mixing strings/ints in WHERE clauses against typed PG columns | cast explicitly, or fix upstream so the param matches the column type |
| Direct sqlite3 | `import sqlite3` / `sqlite3.connect(...)` in runtime modules | route everything through `db_optimizer` |
| Transactions | implicit auto-commit assumptions | use the optimizer's connection/cursor lifecycle |

## Open schema questions — RESOLVED

These were answered 2026-05-10. Decisions baked into the PR plan:

- **Q1 — AUTOINCREMENT**: option (a) — replace with `BIGSERIAL PRIMARY KEY` in all 33 places, no ALTERs to existing prod tables. Fresh-DB bootstrap now works; prod behavior unchanged.
- **Q2 — Booleans-as-INTEGER**: option (b) — this audit only fixes the comparison operators (`= 1` → `= TRUE`). Column-shape audit deferred.
- **Q3 — TEXT-storing-timestamps**: option (a) — discovery script now, append findings to this doc, decide per-table later.

### Q1 — `AUTOINCREMENT` in `CREATE TABLE IF NOT EXISTS` bootstrap statements

Across 14 files we still emit DDL that says:

```sql
CREATE TABLE IF NOT EXISTS foo (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    ...
);
```

On the production PG database these tables already exist, so `IF NOT EXISTS` keeps the bootstrap a no-op today. But:

1. The DDL is **wrong** — if someone bootstraps a fresh PG database (staging, disaster recovery, a new dev DB), every one of these CREATE TABLE will fail with `syntax error at or near "AUTOINCREMENT"`.
2. There's no guarantee the prod tables' `id` columns are actually `BIGSERIAL` / `IDENTITY`. They may have been created by an early Alembic-style run with a different shape, or with a sequence that's now out of sync with `max(id)`.

Proposed direction (need approval before doing):
- **(a) Make the DDL PG-correct.** Replace `INTEGER PRIMARY KEY AUTOINCREMENT` with `BIGSERIAL PRIMARY KEY` in every `CREATE TABLE IF NOT EXISTS`. Existing prod tables untouched. Net effect: no behavior change in prod, but fresh-DB bootstrap now works.
- **(b) Same as (a), PLUS** a one-shot diagnostic script that inspects production PG and reports any `id` column that ISN'T a `BIGSERIAL`/`IDENTITY` (so we can decide whether to ALTER them).
- **(c) Defer entirely.** Mark these as known-warn, no DDL changes; revisit when we adopt Alembic.

### Q2 — Booleans in `INTEGER` columns

Several boolean fields (`is_active`, `granted`, `onboarding_completed`, `enabled`, etc.) are declared as `INTEGER` in the bootstrap DDL but used as booleans throughout the code. This is why we keep hitting `operator does not exist: boolean = integer` errors — the column might actually be `BOOLEAN` in prod PG (created by SQLAlchemy at some point) but our DDL says `INTEGER`.

This is detected indirectly via the `boolean = integer literal` scanner hits — every one of those is a symptom of "code thinks the column is boolean, but our DDL says integer." Today's audit fixes the comparisons. The real question:

- **(a) Standardize on `BOOLEAN`** in DDL and code. Audit the prod schema (script) and decide per-table whether an ALTER is needed.
- **(b) Standardize on `INTEGER 0/1`** everywhere (treat as bit flag). Update code paths that use `= TRUE` / `= FALSE` to use `= 1` / `= 0` — except we know PG rejects that against BOOLEAN columns, so this would only work if the columns truly are INTEGER.
- **(c) Keep current state** — fix the comparison operators (this audit), defer the schema-shape question.

### Q3 — TEXT columns storing timestamps

We already hit this once: `automation_jobs.created_at` is `TEXT` but compared against `TIMESTAMPTZ`. We added `_pg_ts()` casts as a workaround. Same pattern may exist on other tables. Worth a discovery script?

## Process

For each feature area:

1. **Audit** — list every issue under that area from the inventory.
2. **Plan** — for each issue, write the fix sketch in this doc. If schema change is required, flag `🟡` and ASK before proceeding.
3. **Implement** — make the changes; reuse `db_optimizer` helpers (`sql_*`, `_pg_ts`, `sql_true_literal`, etc.) and add new helpers in `core/database_optimization.py` when a pattern recurs.
4. **Smoke test** — `python3 -c "import <module>"` and any existing pytest that covers the file.
5. **PR** — title `pg-audit(<area>): <summary>`, link the inventory rows it closes. Body includes the rows being closed.
6. **Update this doc** — flip status to ✅ and append change-log row.
